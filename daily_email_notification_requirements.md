# 日次イベント検知・メール通知 要件定義

## 目的

売買代金モメンタム条件に新規ヒットした銘柄を毎日検知し、確認用レポートをメールで通知する。

本機能は売買判断を自動化するものではなく、投資候補の発見を補助する通知機能とする。

---

## 前提

既存の売買代金モメンタム検証ロジックを利用する。

既存仕様の主な前提:

* JPX公式一覧から現在上場中の国内株式を取得する
* ETF、ETN、REIT、インフラファンド、出資証券などは除外する
* yfinance から価格データと現在時価総額を取得する
* 取得データは SQLite に保存し、再利用する
* 現在時価総額300億円以上の銘柄を検証対象とする
* TOPIX は `1306.T` を代替ベンチマークとして使用する
* 上場廃止銘柄は本フェーズでは対象外とする

---

## 日次処理の流れ

日次処理は以下の順で実行する。

1. 設定ファイルを読み込む
2. SQLite キャッシュを初期化または接続する
3. JPX公式一覧から現在上場中の株式ユニバースを取得する
4. 現在時価総額を必要に応じて更新する
5. yfinance から当日までの価格データを取得・更新する
6. 最新取引日を基準にイベント検知を実行する
7. 新規ヒット銘柄のみを抽出する
8. CSV と Markdown の日次レポートを出力する
9. ヒット銘柄がある場合、メール通知する
10. 実行結果、通知結果、エラー内容をログに記録する

GitHub Actions で実行する場合、実行環境は毎回初期化されるため、SQLite DB と yfinance キャッシュを永続化または復元できる仕組みを必須とする。
永続化できない場合は毎回全銘柄の価格データを再取得することになり、処理時間、通信量、yfinance 側の制限リスクが高くなるため、本番運用前にキャッシュ復元が正常に機能することを確認する。

日次更新では、保存済み銘柄について5年分を毎回再取得せず、直近の未取得期間または直近日数のみを更新して SQLite に upsert する。
初回構築時のみ5年分を取得し、2回目以降は差分更新を原則とする。

---

## イベント検知条件

既存のイベント条件を利用する。

現在の条件:

* 資金流入比率 ≥ 1.8
* 資金流入比率、売買代金20日平均、75日移動平均条件が5営業日以上継続
* 売買代金20日平均 ≥ 30億円
* 株価 > 75日移動平均
* 75日移動平均が上向き
* イベント日の調整後終値 > 前営業日の調整後終値
* イベント日の調整後終値 > 5営業日前の調整後終値
* 同一銘柄はイベント発生後60営業日は再登録しない

日次通知では、最新取引日に新規イベントとして確定した銘柄を通知対象とする。

また、新規イベントだけでなく、過去にヒットした監視中銘柄も通知対象に含める。

監視中銘柄の対象期間は、イベント日から63営業日以内とする。

通知上の区分:

* 本日新規ヒット
* 監視中: ヒットから1〜21営業日
* 監視中: ヒットから22〜63営業日

ヒットから64営業日以上経過した銘柄は、日次メール本文には掲載しない。ただし、SQLiteやCSVには履歴として保持する。

ヒットからの日数は、カレンダー日数ではなく営業日数で算出する。

---

## 新規ヒット判定

同じイベントを複数回通知しないため、通知済みイベントを永続管理する。

通知済み判定キー:

* 銘柄コード
* イベント日
* シグナル条件バージョン、または設定ハッシュ

同一銘柄・同一イベント日・同一条件で通知済みの場合は再通知しない。

設定変更により条件が変わった場合は、過去イベントの再通知を避けるため、以下のどちらかを選択可能にする。

* 設定変更後も同一銘柄・同一イベント日は再通知しない
* 設定ハッシュが変わった場合は別イベントとして扱う

初期実装では、同一銘柄・同一イベント日は再通知しない方式を採用する。

### 初回実行時の扱い

通知履歴が存在しない初回実行時は、最新取引日を基準として過去63営業日以内に発生したイベントを通知対象に含める。

初回メールでの分類:

* 最新取引日にヒットしたイベントは「本日新規ヒット」として扱う
* 1〜21営業日前にヒットしたイベントは「監視中: ヒットから1〜21営業日」として扱う
* 22〜63営業日前にヒットしたイベントは「監視中: ヒットから22〜63営業日」として扱う
* 64営業日以上前にヒットしたイベントは通知しない

初回通知後、通知対象になったイベントは通知履歴に登録する。

2回目以降の実行では、最新取引日に新しくヒットしたイベントのみを新規ヒットとして扱い、過去63営業日以内の既存イベントは監視中として通知する。

## メール通知

メール送信には Gmail の SMTP を使用する。
初期実装では Gmail のアプリパスワードを GitHub Actions Secrets またはローカル `.env` から読み込む。
通常の Google アカウントパスワードを設定ファイルへ直接保存してはならない。

### 通知タイミング

日次処理で新規ヒット銘柄が1件以上ある場合にメールを送信する。

新規ヒット銘柄が0件でも、監視中銘柄が1件以上ある場合はメールを送信する。

新規ヒット銘柄も監視中銘柄も0件の場合は、既定ではメール送信しない。

オプションで、0件でも実行完了メールを送信できるようにする。

### メール件名

例:

```text
[売買代金モメンタム] 2026-07-03 新規2件 / 監視中8件
```

### メール本文

メール本文は日本語で出力する。

メール本文は、Gmail やスマートフォンでの可読性を優先し、Markdown の表形式ではなく銘柄ごとのブロック形式で出力する。

本文に含める項目:

* 実行日時
* 対象最新取引日
* 検証対象銘柄数
* 新規ヒット件数
* 監視中件数
* ヒット銘柄一覧

本文に含めない項目:

* 監視対象期間の説明
* レポート出力先
* 売買推奨ではない旨の注記

本日新規ヒットに含める項目:

* 銘柄コード
* 銘柄名
* イベント日
* ヒットからの経過営業日数
* 調整後終値
* 売買代金20日平均
* 資金流入比率
* 75日移動平均

監視中銘柄に含める項目:

* 銘柄コード
* 銘柄名
* イベント日
* ヒットからの経過営業日数
* 調整後終値
* イベント日比
* 資金流入比率

本文では、本日新規ヒットと監視中銘柄を分けて表示する。

本文構成:

1. ヘッダー
2. 本日新規ヒット
3. 監視中: ヒットから1〜21営業日
4. 監視中: ヒットから22〜63営業日

初期実装では添付ファイルは必須としない。

ただし、以下のファイルを添付またはリンクできる設計にする。

* 日次イベントCSV
* 日次Markdownレポート

## 出力ファイル

日次処理では、日付別フォルダにレポートを出力する。

例:

```text
reports/daily/2026-07-03/
```

出力ファイル:

* `events.csv`
* `report.md`
* `run.log`

必要に応じて以下も出力する。

* `all_detected_events.csv`
* `notification_history.csv`

## 設定項目

既存設定に加えて、日次通知用の設定を追加する。

### 実行設定

* 日次処理の有効・無効
* レポート出力先
* 0件時メール送信の有効・無効
* 通知済みイベント管理DBまたはテーブル

### メール設定

* SMTPホスト（Gmailの場合は `smtp.gmail.com`）
* SMTPポート（Gmail + STARTTLS の場合は `587`）
* TLS/SSLの使用有無
* SMTPユーザー名
* SMTPパスワード、または Gmail アプリパスワード
* 送信元メールアドレス
* 送信先メールアドレス
* CC
* BCC

SMTPパスワードなどの秘密情報は設定ファイルに直接書かず、環境変数または `.env` ファイルから読み込む。

GitHub Actions で使用する Secrets の想定:

* `GMAIL_SMTP_USER`
* `GMAIL_APP_PASSWORD`
* `MAIL_FROM`
* `MAIL_TO`
* `MAIL_CC`
* `MAIL_BCC`

`MAIL_FROM` と `MAIL_TO` は省略可能とする。
未指定の場合は、どちらも `GMAIL_SMTP_USER` と同じメールアドレスを使用する。

## スケジューリング

初期実装では、アプリケーション自身に常駐スケジューラを持たせず、GitHub Actions からCLIを毎日実行する。

GitHub Actions は以下の2種類の実行方法を用意する。

* cron による平日定期実行
* `workflow_dispatch` による手動実行

実行タイミングは、東証の当日データが yfinance で取得可能になる時間帯を考慮し、平日夕方以降または夜間とする。
GitHub Actions の cron は UTC で指定するため、20:00 JST 実行の場合は 11:00 UTC を指定する。

例:

```text
平日 20:00 JST
GitHub Actions cron: 0 11 * * 1-5
```

祝日・休場日は、最新取引日が前回通知済みの日付と同じ場合、新規イベントなしとして扱う。

GitHub Actions 上では、SQLite DB と yfinance キャッシュを `actions/cache` などで復元し、実行後に更新済みキャッシュを保存する。
キャッシュ復元に失敗した場合でも処理は実行可能とするが、本番運用では警告としてログに残す。

---

## エラーハンドリング

以下のエラーはログに記録する。

* JPX一覧取得失敗
* yfinance価格データ取得失敗
* 現在時価総額取得失敗
* SQLite書き込み失敗
* イベント検知失敗
* メール送信失敗

一部銘柄の取得に失敗しても、全体処理は可能な限り継続する。

メール送信に失敗した場合でも、日次レポートと通知候補CSVは保存する。

---

## ログ

日次処理ごとにログを出力する。

ログに含める項目:

* 実行開始日時
* 実行終了日時
* 対象最新取引日
* 取得対象銘柄数
* 価格データ更新成功件数
* 価格データ更新失敗件数
* 新規ヒット件数
* メール送信結果
* エラー詳細

## 運用上の注意

本機能は現在上場中の銘柄を対象とするため、上場廃止銘柄は含まれない。

現在時価総額フィルタは、現在時点の情報を使う近似フィルタであり、イベント日時点の厳密な時価総額条件ではない。

yfinance のデータ更新タイミングや欠損により、同じ日でも実行時刻によって結果が変わる可能性がある。

通知された銘柄は売買推奨ではなく、追加確認の候補として扱う。

---

## 初期実装の対象範囲

初期実装では以下を対象とする。

* CLIによる日次実行
* SQLiteキャッシュの再利用
* GitHub Actions による定期実行
* GitHub Actions 上での SQLite DB と yfinance キャッシュの復元・保存
* Gmail SMTP によるメール送信
* 直近未取得期間または直近日数のみの価格データ差分更新
* 最新取引日の新規イベント検知
* 初回実行時の過去63営業日以内イベント通知
* 通知済みイベントの重複排除
* 日本語Markdown日次レポート出力
* 実行ログ出力

---

## 初期実装では対象外

以下は初期実装では対象外とする。

* GUI
* Webダッシュボード
* 証券会社API連携
* 自動売買
* ポジション管理
* リアルタイム監視
* LINE、Slack、Discordなどメール以外の通知
* 複数戦略の同時管理
