# 売買代金モメンタム投資 実装計画

## 前提

本計画は `spec.md` に基づき、売買代金の継続的な増加が将来の株価上昇の先行指標となるかを検証するための実装方針をまとめる。

この段階では、エントリー、エグジット、損切、利確、ポジション管理、資金管理などの売買ルールは実装対象外とする。目的は、資金流入イベントの検出と、その後のリターン・ベンチマーク比較を定量的に評価することである。

## 実装ステップ

### 1. 最小構成の設計とプロジェクト骨格作成

Python ベースで、設定、データ取得、指標計算、イベント検出、評価集計、レポート出力を分けた構成にする。

パラメータはコード直書きにせず、設定ファイルで変更できるようにする。対象パラメータは、売買代金短期平均日数、売買代金長期平均日数、資金流入比率、継続営業日数、最低売買代金、移動平均期間、再登録禁止期間、バックテスト期間、現在時価総額フィルタの有効・無効、現在時価総額下限とする。

現在時点の時価総額は、過去イベント日時点の厳密な条件ではなく、低位株や極端な小型株を除外するためのユニバースフィルタとして扱う。yfinance から取得できる現在時価総額を使い、設定ファイルで簡単に有効・無効を切り替えられるようにする。

現在時価総額フィルタにはルックアヘッド性があるため、イベント条件そのものとは分けて扱い、最終レポートと README に「現在時点の時価総額による近似的なユニバースフィルタ」であることを明記する。

品質確保のため、各処理の責務を明確にし、後続ステップで単体テストしやすい構造にする。

### 2. データ取得・正規化・再利用レイヤーの実装

yfinance を使用して、個別銘柄、TOPIX、日経平均の OHLCV データを取得する。yfinance の `download` API は `period=5y` と `interval=1d` を指定できるため、5年分の日足 OHLCV 取得を基本方針とする。ただし、Yahoo Finance 側の銘柄カバレッジ、上場廃止銘柄、ティッカー形式、欠損により、全銘柄で完全な5年分が取得できるとは限らない前提で扱う。

取得したデータは、日付、銘柄コード、始値、高値、安値、調整後終値、出来高、売買代金を持つ統一形式に正規化する。売買代金は `調整後終値 × 出来高` で近似する。

取得失敗、欠損、営業日ズレ、データ不足はログに残し、後から検証できるようにする。特に4000銘柄規模では、取得成功銘柄数、取得失敗銘柄数、期間不足銘柄数、ベンチマーク取得可否をレポートに含める。

一度取得したデータはローカルに保存し、以後のバックテストでは再利用できるようにする。これにより、資金流入比率、継続営業日数、移動平均期間などのアルゴリズムやパラメータを変更した場合でも、毎回数年分の価格データを再取得せずに検証できる。

データ取得処理と分析処理は分離する。CLI では、データ取得のみを行うモード、保存済みデータのみで分析するモード、必要な不足分だけを追加取得して分析するモードを用意する。

保存済みデータには、銘柄コード、取得期間、取得日時、データソース、調整後終値の使用有無、取得成功・失敗状態、現在時価総額などのメタ情報を付与する。実行時には、要求されたバックテスト期間をローカルデータが満たしているかを確認し、不足している期間だけを追加取得できるようにする。

保存形式は、4000銘柄規模での再利用を前提に SQLite を基本とする。価格データ、ベンチマークデータ、銘柄メタ情報、現在時価総額、取得履歴をテーブルとして管理し、銘柄コードと日付で検索・更新しやすい構成にする。

CSV と Markdown は分析結果の出力形式として使用する。価格データの主保存先は SQLite とし、必要に応じて検証用や外部確認用に CSV エクスポートできるようにする。

### 3. 指標計算とイベント検出の実装

売買代金短期平均、売買代金長期平均、資金流入比率、移動平均、移動平均の傾きを計算する。

イベント条件は、資金流入比率、継続営業日数、最低売買代金、株価の移動平均超え、移動平均の上向きをすべて満たすこととする。条件を初めて満たした営業日をイベント日として記録する。

現在時価総額フィルタはイベント条件ではなく、分析対象銘柄を絞り込むユニバースフィルタとしてイベント検出前に適用する。設定ファイルで無効化できるようにし、フィルタ有無で結果を比較できるようにする。

条件継続中は再登録せず、同一銘柄についてはイベント発生後の再登録禁止期間を適用する。上場廃止銘柄も可能な限り対象に含める。

品質確保のため、人工データを使った単体テストを作成し、イベント発生日、重複登録抑制、再登録禁止期間の挙動を確認する。

### 4. イベント後リターン・リスク・ベンチマーク比較の集計

イベント日を基準に、21営業日後、63営業日後、126営業日後、252営業日後のリターンを算出する。

各評価期間について、基本統計、リターン分布、リスク指標、ベンチマーク比較を集計する。

出力する主な指標は以下とする。

| 区分 | 出力項目 |
| --- | --- |
| 基本統計 | サンプル数、平均リターン、中央値リターン、最大リターン、最小リターン、標準偏差、勝率 |
| リターン分布 | +10%以上到達率、+20%以上到達率、+30%以上到達率、+50%以上到達率、+100%以上到達率 |
| リスク指標 | 最大上昇率、最大下落率、最大ドローダウン、平均ドローダウン |
| ベンチマーク比較 | TOPIXリターン、日経平均リターン、TOPIX超過リターン、日経平均超過リターン |

計算結果については、イベント単位の詳細結果と、評価期間ごとの集計結果を分けて扱う。

### 5. 実行CLI・出力・README・検証ワークフロー整備

利用者が自分で実行できるように、対象銘柄リスト、バックテスト期間、設定ファイル、出力先を指定できる CLI を用意する。

CLI には、少なくともデータ取得、キャッシュ更新、バックテスト実行、レポート生成の役割を持たせる。保存済みデータを使う場合と、必要に応じて追加取得する場合を選べるようにする。

結果は機械的な再利用に向いた CSV と、人間が確認しやすい Markdown の両方で出力する。Markdown レポートは必ず日本語で出力する。Markdown レポートでは table を活用し、評価期間ごとの統計、ベンチマーク比較、主なイベント一覧、データ取得・除外に関する注意点を見やすく整理する。

Markdown レポートには、使用した設定値、現在時価総額フィルタの有効・無効、現在時価総額下限、ユニバース対象銘柄数、除外銘柄数を含める。現在時価総額フィルタを使った場合は、現在時点の情報を使った近似フィルタであることを日本語で明記する。

README を作成し、セットアップ、依存関係、実行方法、設定項目、入力ファイル形式、出力ファイルの見方、データキャッシュの仕組み、現在時価総額フィルタの考え方と切り替え方法、再取得が必要になるケース、よくある失敗例を記載する。

品質確認として、まず少数銘柄・短期間で動作確認し、その後に過去1年、3年、5年の順で実行する。イベント件数、重複登録、欠損データ、ベンチマーク比較、Markdown レポートの読みやすさを確認してから完了とする。

## 実装時の注意点

* 本フェーズでは売買判断の良し悪しではなく、資金流入イベントの統計的な有効性を確認する。
* yfinance 由来の欠損、取得失敗、期間不足、上場廃止銘柄の扱いは結果に影響するため、ログとレポートに残す。
* 現在時価総額フィルタは低位株・極端な小型株を除外するためのユニバースフィルタとして扱い、イベント条件とは分ける。
* 現在時価総額フィルタにはルックアヘッド性があるため、設定ファイルで無効化できるようにし、最終レポートと README に明記する。
* データ取得と分析を分離し、アルゴリズム変更時に保存済みデータを再利用できるようにする。
* 保存済みデータを使う場合でも、対象期間の不足、銘柄コードの不足、ベンチマークデータの不足は実行前に検出する。
* ベンチマークとの比較は同じイベント日、同じ評価期間で行う。
* Markdown レポートは必ず日本語で出力し、後から人間が読み返して判断できるように、数値だけでなく実行条件も含める。
* 実装後の最初の検証は小さなデータセットで行い、問題がなければ対象期間と対象銘柄を広げる。
