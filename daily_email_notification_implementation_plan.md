# 日次イベント検知・Gmail通知 実装計画

## 前提

本計画は `daily_email_notification_requirements.md` を実現するための実装計画である。
前回の反省を踏まえ、単にメール送信を追加するのではなく、以下を一体として実装対象に含める。

* 日次差分更新
* SQLite DB と yfinance キャッシュの再利用
* 通知履歴の永続管理
* 初回実行時の過去63営業日以内イベント通知
* Gmail SMTP 送信
* GitHub Actions 定期実行
* 日本語の日次MarkdownレポートとCSV出力
* README による運用手順

## 実装ステップ

### Step 1: 設定・DB・永続化設計を固める

日次通知用の設定を `config.example.json` と設定読み込み処理へ追加する。
追加対象は、日次処理の有効化、通知対象期間63営業日、0件メール送信、日次レポート出力先、Gmail SMTP、環境変数名、GitHub Actions 実行時のキャッシュ前提である。

SQLite には通知履歴を管理するテーブルを追加する。
少なくとも、銘柄コード、イベント日、通知日時、通知区分、シグナル条件識別子を保存し、同一銘柄・同一イベント日の重複通知を防ぐ。
必要であれば日次実行ログ用テーブルも追加するが、巨大な設計変更は避ける。

品質確認:

* 設定ファイルの読み込みテスト
* 既存設定との後方互換テスト
* 通知履歴テーブルの初期化テスト
* 空DBでも既存バックテストが壊れないことの確認

### Step 2: 価格データの日次差分更新を実装する

既存の `fetch-data` は保存済み銘柄をスキップするため、そのままでは日次更新に使えない。
日次処理では、初回は5年分を取得し、2回目以降は各銘柄の最新保存日から現在まで、または設定した直近日数だけを再取得して SQLite に upsert する。

GitHub Actions では実行環境が毎回初期化されるため、SQLite DB と yfinance キャッシュを復元できることを前提にする。
キャッシュが復元できた場合は差分更新、復元できなかった場合は初回構築として扱い、ログに警告を出す。

品質確認:

* 既存データありの場合に全期間再取得しないこと
* 既存データなしの場合に初回取得として動くこと
* 直近データの重複 upsert が壊れないこと
* 一部銘柄の取得失敗時に全体処理が継続すること
* `--limit` を使った小規模スモークテスト

### Step 3: 日次イベント抽出・初回63営業日通知・レポート出力を実装する

既存のイベント検知ロジックを再利用し、最新取引日を基準に日次通知対象を抽出する。
通知履歴が空の初回実行では、過去63営業日以内のイベントを通知対象に含める。
2回目以降は、最新取引日の未通知イベントを「本日新規ヒット」とし、過去63営業日以内の既存イベントを監視中として通知する。

日次レポートは `reports/daily/YYYY-MM-DD/` に出力する。
人間が確認しやすい日本語Markdownと、機械処理しやすいCSVの両方を出力する。
Markdown はメール本文と同様に、本日新規ヒット、監視中1〜21営業日、監視中22〜63営業日に分ける。

品質確認:

* 初回実行で過去63営業日以内が通知対象になること
* 64営業日以上前のイベントがメール本文から除外されること
* 2回目以降に同一イベントが新規扱いで再通知されないこと
* 経過日数がカレンダー日数ではなく営業日数で算出されること
* イベント0件、監視中0件、監視中のみ、新規のみ、新規+監視中の各ケース

### Step 4: Gmail送信・CLI・GitHub Actionsを接続する

Gmail SMTP 送信用の処理を追加する。
秘密情報は設定ファイルに保存せず、ローカルでは `.env`、GitHub Actions では Secrets から読み込む。
メール本文はMarkdownテーブルではなく、Gmailやスマートフォンで読みやすいブロック形式で生成する。

CLI には日次実行用コマンドを追加する。
想定コマンドは `daily-notify` とし、ドライラン、メール送信有無、銘柄数制限、基準日の指定、価格更新有無を切り替えられるようにする。

GitHub Actions には、平日20:00 JST相当の cron と手動実行を追加する。
Actions では SQLite DB と yfinance キャッシュを復元・保存し、Secrets から Gmail 認証情報を渡す。

品質確認:

* メール本文生成の単体テスト
* SMTP送信をモックしたテスト
* `--dry-run` でメールを送らず日次レポートだけ出力できること
* GitHub Actions の YAML 構文確認
* Secrets 未設定時に分かりやすいエラーになること

### Step 5: README整備・総合テスト・運用前確認を行う

README に、ローカル実行、初回データ取得、日次ドライラン、Gmail設定、GitHub Actions Secrets、手動実行、本番定期実行、トラブルシュートを追記する。
ユーザーが運用できるように、実行コマンドと確認すべき出力ファイルを明記する。

総合テストは段階的に実施する。
まず `--limit` 付きで日次ドライランを実行し、次に既存SQLiteを使った全件ドライランを行う。
最後に、メール送信を有効化した小規模テストを行い、本文、件名、通知履歴、日次レポートが一致していることを確認する。

品質確認:

* 既存バックテストテストが通ること
* 日次通知の新規テストが通ること
* 小規模ドライランでCSVとMarkdownが出力されること
* 通知履歴により2回目実行で新規再通知されないこと
* READMEだけを見て初回運用準備ができること

## 実装時に特に注意する点

* GitHub Actions の実行環境は永続化されないため、キャッシュ復元を実装対象から外さない。
* 既存の `fetch-data` のスキップ挙動だけでは日次更新にならないため、差分更新を必ず実装する。
* 初回実行では最新取引日だけでなく、過去63営業日以内の既存イベントも通知する。
* 日次通知は全上場銘柄を対象にしたスクリーニングであり、ユーザー指定の銘柄CSVを前提にしない。
* メール本文は日本語で、表形式ではなくブロック形式にする。
* Markdownレポートも必ず日本語で出力する。
* Gmail の認証情報は GitHub Secrets または `.env` に置き、設定ファイルには直接書かない。
* 祝日や休場日は、最新取引日が進んでいなければ新規イベントなしとして扱う。

* 2回目以降に同一イベントを新規として重複通知しない

* 日次CSVと日本語Markdownレポートが出力される
* README に運用手順が記載されている
* 既存テストと追加テストが成功している
